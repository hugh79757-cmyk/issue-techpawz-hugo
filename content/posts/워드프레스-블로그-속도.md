워드프레스 블로그 속도 측정을 해보면 결과는 좋은데, 체감 속도는 느린 경우 - 워드프레스 블로그 운영자라면 한 번쯤 이런 경험이 있으실 겁니다. WP Rocket 같은 캐시 플러그인도 설치하고, 이미지 최적화까지 마쳤는데, 페이지 전환 속도는 여전히 더딘 느낌. 캐시 플러그인을 제대로 설치했다면 [Google PageSpeed Insights](https://pagespeed.web.dev/?hl=ko)에서 측정해보면 성능 점수는 90점 이상이 나옵니다. 하지만 실제 사용 중에는 5~8초씩 멈칫하는 이질감이 생길 때가 있어요. 얼마 전까지 [패스트코맷](https://www.fastcomet.com/)을 이용하던 워드프레스에서 그런 현상이 있었습니다.

이 글에서는 속도 측정 점수는 높지만 **체감 속도는 느린 워드프레스 사이트**에 대해, 직접 점검할 수 있는 **5가지 핵심 체크포인트**를 알려드립니다.

## 워드프레스 블로그 속도가 느릴 때

### 서버 성능이 진짜 발목을 잡고 있는가?

가장 먼저 확인해야 할 것은 **호스팅 서버의 응답속도**입니다.  
PageSpeed 점수는 캐시 파일을 기준으로 테스트되기 때문에, 실제 방문자의 첫 페이지 로딩 경험과는 다를 수 있습니다.

- **공유 호스팅**이라면 다른 사용자의 영향으로 순간적인 부하가 생길 수 있습니다.
- 서버 응답 시간이 600ms 이상이라면, 성능 저하를 체감하게 됩니다.

**해결 방법:**

- PHP 8.1 이상으로 업그레이드
- LiteSpeed 서버 기반 호스팅으로 이전 고려
- FastComet, Cloudways 등 빠른 호스팅 서비스 이용

### 너무 많은 플러그인 사용

기능 확장에 유용한 플러그인이지만, **비효율적인 플러그인**은 사이트를 느리게 만듭니다.  
다음과 같은 플러그인은 로딩 지연의 주범이 될 수 있습니다.

- 외부 JS를 불러오는 SNS 공유 플러그인
- 각종 통계, 방문자 분석 플러그인
- 시각효과를 위한 슬라이더, 애니메이션 플러그인

**해결 방법:**

- `Query Monitor` 플러그인으로 실행 속도 확인
- 느린 플러그인은 비활성화 후 대체 솔루션 검토

### TTFB(Time To First Byte) 지연 문제

TTFB는 사용자가 브라우저에서 요청을 보낸 뒤, **첫 바이트가 도착하기까지의 시간**입니다.   
페이지 전환이 느리다고 느껴질 때, 이 값이 길다면 서버가 응답을 너무 늦게 보내고 있는 것입니다.

**해결 방법:**

- [Cloudflare](https://www.cloudflare.com/ko-kr/) 같은 CDN 서비스 도입
- 오리진 서버 위치 확인 → 사용자의 지역과 가까운 서버로 이관
- cloudflare를 이용할 때 서브도메인을 위해 지정한 CNAME, A 의 세부 내용은 자동으로 옮겨지지 않습니다. 다시 설정해야 합니다. 옮기기 전에 서브도메인 지정을 새로하기 위한 기록을 남겨두세요.

### 외부 스크립트 요청이 많음

Google Fonts, YouTube embed, 외부 광고, 분석 도구 등의 JS 로딩은 **페이지 렌더링을 지연**시킵니다.

**해결 방법:**

- 외부 JS는 `async` 혹은 `defer` 로 비동기 처리
- 폰트는 사이트 내에 직접 호스팅하거나 preload 설정
- 사용하지 않는 외부 코드 제거

### 캐시 설정이 누락되었거나 충돌 중

WP Rocket, LiteSpeed Cache 등 캐시 플러그인을 사용하더라도, **캐시 예외 설정**이나 **충돌**로 인해 일부 페이지에서는 효과가 없습니다.

**해결 방법:**

- 캐시 플러그인의 설정 탭에서 `로그인 페이지 제외`, `쿼리 문자열 무시` 등 확인
- 타 캐시 시스템(호스팅 자체 캐시, Cloudflare 캐시 등)과 충돌 여부 점검
- 캐시 삭제 후 재생성 실행
